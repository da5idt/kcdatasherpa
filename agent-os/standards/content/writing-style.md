# Writing Style

## Post Types

### Link / Commentary
Use when reacting to an external article, announcement, or resource.

- Always include at least one personal take or lesson — don't just summarize
- Ends with the source link
- No H2 headers required; keep it tight

### Original Analysis
Use when you have an original argument or framework to build — not just a reaction.

- Use when the point can't be made in a few paragraphs and you're building something, not reacting
- Uses H2 headers to organize sections
- Ends with a References section (source name + link)

## Voice

- **Direct.** State the point, then explain. No throat-clearing.
- **Practitioner POV.** Written from lived experience in data engineering and leadership.
- **First person.** "I", "we" — not "organizations should" or "teams must."
- **Short sentences.** Fragments for emphasis are fine. ("Not at scale. Not reliably.")
- **Personal anecdotes when genuinely relevant** — not forced. Only when they add a real lesson or credibility.

**Avoid:**
- Hype words: "revolutionary", "game-changing", "unprecedented", "transformative"
- Passive hedging: "some might argue", "it could be said", "many believe"
- Academic tone: no white-paper abstracts, no excessive citations
- Listicles without substance: bullets support an argument, they don't replace one

## Post Structure

- **Headers:** Use H2 (`##`) for sections. Never use H1 (`#`) in post body — the layout renders the front matter `title` as H1.
- **Original analysis posts:** End with a References section (source name, title as link).
- **Link/commentary posts:** End with the source link inline or as a plain labeled link — no formal References section needed.
- **Blockquotes:** Use sparingly for direct quotes or key emphasis.
- **Opening:** No rigid rule — open however fits the post. Avoid generic intros ("In today's world...").
